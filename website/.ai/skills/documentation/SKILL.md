# Documentation

## Purpose

Make the project easy to understand and safe to change. Doc changes ship with the code change they describe.

## When to use

- Exported functions, components, routes.
- Non-obvious decisions (use an ADR).
- User-facing setup or run instructions (`README.md`).
- AI-system conventions (`.ai/` and `docs/ai-system/`).

## Procedure

1. Decide the canonical location (see [`../../rules/documentation.md`](../../rules/documentation.md)). One place per topic.
2. Write the smallest useful text: purpose, inputs, outputs, side effects.
3. Prefer links to duplication.
4. Show a concrete example if the API is non-trivial.
5. Update docs in the same PR as the code.

## Quality bar

- [ ] Short, active voice, concrete examples.
- [ ] Placed in the canonical location; no duplicates.
- [ ] No narration of obvious code.
- [ ] Internal links resolve.

## Anti-patterns

- Autogenerated walls of text with no curation.
- Parallel copies of the same content in multiple files.
- Comments that narrate what code already says.
